1. Filosofi & Core System
Identitas: Nama produk Space.in. Kesan modern, action-oriented, dan relevan untuk Gen Z.

Konsep Dasar: Bukan sekadar aplikasi formulir digital, melainkan Workflow Engine. Aturan birokrasi, urutan persetujuan, dan syarat dokumen disimpan sebagai data di database, bukan di-hardcode dalam script program.

Prinsip Keamanan:

Anti-Overlap: Menggunakan logika Soft-Lock (kunci sementara saat pengisian) dan Hard-Lock (kunci permanen setelah disetujui).

Atomic Transaction: Mencegah dua orang mem-booking ruangan yang sama di milidetik yang sama (menggunakan lockForUpdate di Laravel).

2. Arsitektur Database (PostgreSQL)
Kita menggunakan PostgreSQL dengan Primary Key tipe UUID.

A. Perbaikan Struktur (Crucial Fixes):

Hierarki Unit (units):

Wajib tambah kolom parent_id dan level ('Pusat', 'Jurusan', 'Organisasi').

Tujuannya agar sistem tahu struktur pohon (HMTI adalah anak Jurusan TI, Jurusan TI adalah anak Pusat).

Kepemilikan Ruangan (rooms):

Wajib tambah kolom unit_id.

Tujuannya untuk validasi otoritas: Siapa yang berhak mengatur jadwal ruangan ini? (Misal: Auditorium milik Pusat, Lab Komputer milik Jurusan).

Logika Approver (workflow_steps):

Kolom requires_attachment (Boolean).

Jika true, tombol "Setuju" milik pejabat tersebut terkunci sampai dia mengunggah dokumen (misal: Surat Balasan/Dispensasi).

3. Logika Bisnis & Workflow
Alur kerja dibagi menjadi 4 Fase Tegas:

Fase Setup (Admin Unit):

Admin menyusun "rantai" pejabat dan syarat dokumen.

Aturan Emas: Admin Organisasi (BEM/HMTI) tidak boleh mengatur workflow untuk ruangan milik pihak lain (Pusat/Jurusan). Mereka harus tunduk pada Workflow pemilik ruangan.

Fase Pengajuan (Peminjam):

Pilih jadwal -> Auto Check Bentrok -> Soft-Lock.

Upload dokumen syarat -> Submit.

Fase Persetujuan (Approver):

Setuju: Lanjut ke pejabat berikutnya.

Tolak: Wajib input alasan (Catatan Revisi). Status booking batal/revisi.

Fitur Digital: Tidak ada langkah "Stempel" terpisah. Tanda tangan digital dan stempel ditempel otomatis oleh sistem saat klik "Approve".

Fase Finalisasi (Sistem):

Semua setuju -> Status ubah ke Hard-Lock.

Terbit Surat Izin PDF dengan QR Code Validasi.

4. Manajemen Hak Akses (Role vs Scope)
Kita membedakan antara "Apa yang bisa kamu lakukan" (Role) dengan "Di mana kekuasaanmu" (Scope).

SuperAdmin (Pusat):

Menguasai Master Data Global (Gedung, Ruangan).

Menentukan unit_id (kepemilikan) setiap ruangan.

Mengatur Libur Nasional (Global Blocking).

Admin Unit (Jurusan/Organisasi):

Hanya bisa mengelola data di wilayahnya sendiri (filter by parent_id).

Mengatur jadwal/maintenance ruangan miliknya sendiri.

Approver:

Hanya bisa melihat/aksi pada dokumen yang sampai di "meja"-nya.

5. Strategi Operasional (Handling Kasus Nyata)
Solusi cerdas tanpa menambah tabel/role baru:

Blokir Jadwal / Maintenance:

Dilakukan oleh Admin Unit Pemilik Ruangan.

Caranya: Membuat transaksi booking biasa dengan nama event "Maintenance".

Sistem mem-bypass approval dan langsung Hard-Lock.

Libur Nasional:

Metode Hybrid: Sistem menarik API Kalender Nasional -> SuperAdmin melakukan review/edit -> Sistem men-generate booking massal (Global Blocking) ke tabel bookings.

6. Catatan Teknis Laravel (Implementation Guide)

Strict Typing: Validasi input harus ketat karena PostgreSQL tidak menoleransi tipe data yang salah (string vs integer).

Seeding Strategy: Buat seeder untuk data hierarki (Pusat -> Jurusan -> Organisasi) agar saat demo presentasi, logika filternya terlihat jelas.

Ini adalah rangkuman final. Jika kamu memegang teguh dokumen ini, sistem Space.in akan menjadi aplikasi level Enterprise, bukan sekadar tugas kuliah. Siap eksekusi?

Anda berkata
maaf bro, aku mau balik ..nah apakah superAdmin yang input manual building dan roomsnya

Ya, betul. SuperAdmin yang melakukan input data buildings (gedung) dan rooms (ruangan) ke dalam sistem.

Ini alasannya secara logis:

Otoritas Master Data: Gedung dan ruangan adalah aset fisik kampus (Master Data Global). Admin level bawah (Jurusan/Organisasi) tidak boleh punya hak untuk "menciptakan" atau "menghapus" ruangan di dalam sistem sembarangan.

Penentuan Kepemilikan: Saat SuperAdmin menginput data ruangan, di detik itulah SuperAdmin mengunci kolom unit_id. SuperAdmin yang menentukan: "Auditorium ini milik Pusat" atau "Lab Komputer ini milik Jurusan TI".

Secara Teknis di Lapangan:
SuperAdmin memang yang menginput, tapi tidak harus selalu mengetik manual satu per satu lewat form. Untuk efisiensi, kamu bisa membuatkan fitur Import Excel/CSV di dashboard SuperAdmin agar mereka bisa mengunggah ratusan data ruangan sekaligus dalam 1 detik.

Intinya: Otoritas pembuatan data fisik 100% ada di tangan SuperAdmin. Admin Unit di bawahnya cuma menerima "kunci" dan memakai ruangannya. Jelas?
